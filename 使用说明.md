# Houdini MCP 使用说明

## 新架构

当前项目已经改成两层：

1. `后台服务 daemon`
   常驻进程，负责 GUI 连接和 Houdini 操作执行

2. `Codex MCP bridge`
   由 `Codex` 启动，只负责把工具调用转发给后台服务

所以现在：

- GUI 连的是后台服务
- `Codex` 连的是 MCP bridge
- 两者不再争抢同一个短生命周期进程

## 正确使用方式

### 在 Codex 中使用

1. 配置 `C:/Users/wepie/.codex/config.toml`
   `houdini_mcp` 的 bridge 建议用普通 `python` 启动，不要用 `hython`
2. 重启 `Codex`
3. 让 `Codex` 正常加载 `houdini_mcp`

### 打开控制面板

直接运行：

```bash
python run_gui.py
```

或者双击：

```text
启动GUI.bat
```

GUI 会先确保后台服务存在，再连接。

### 单独启动后台服务

如果你想先验证后台服务本身，可以直接运行：

```text
启动后台服务.bat
```

### 做健康检查

可以运行：

```text
验证Codex配置.bat
```

它会检查：

1. `Codex` 配置是否存在 `houdini_mcp`
2. 后台服务状态文件是否存在
3. 后台服务和 bridge 进程是否还活着
4. 日志目录是否正常生成

## 关闭行为

### 关闭 Codex

不会强制要求后台服务立刻一起退出。

这是新架构的目的之一：避免 `Codex` 的 MCP 子进程生命周期影响 GUI。

### 关闭 GUI

只会断开 GUI 自己，不会自动杀后台服务。

### 停止后台服务

可以在 GUI 里点“停止服务器”。

## 日志位置

- 后台服务日志：
  `C:/Users/wepie/.mcp_logs/houdini-mcp-daemon/`
- Codex bridge 日志：
  `C:/Users/wepie/.mcp_logs/houdini-mcp-bridge/`

## 运行状态文件

- `C:/Users/wepie/.codex/mcp/houdini-mcp/.running.lock`
- `C:/Users/wepie/.codex/mcp/houdini-mcp/ws_port.json`
- `C:/Users/wepie/.codex/mcp/houdini-mcp/ws_port_<pid>.json`

## 什么时候看这些文件

如果 GUI 提示断开连接，先看：

1. 后台服务日志有没有新文件
2. `ws_port.json` 是否存在
3. `ws_port_<pid>.json` 对应的 PID 是否还活着
