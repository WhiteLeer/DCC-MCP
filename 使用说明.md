# DCC MCP 使用说明（统一版）

## 1. 架构说明

每个 DCC（Houdini/Maya/Blender/Substance）都使用同一架构：

1. `daemon_server`
常驻后台，持有真实 DCC 会话和状态。

2. `server_with_gui`
MCP bridge，只负责转发工具调用到 daemon。

因此：
- GUI 管理 daemon
- MCP 客户端调用 bridge
- 两者互不抢占短生命周期进程

## 2. 推荐启动方式

优先使用统一面板（默认不会自动弹出）：

```bash
python run_unified_gui.py
```

或双击：

```text
启动DCC-MCP.bat
```

在统一面板里按需使用：
- 模块开关（启动/关闭 daemon）
- 心跳检测（运行中/未响应/未启动/关闭）
- 日志查看（打开时自动加载历史）

## 3. 客户端配置建议

在客户端配置中，bridge 路径统一使用：

`C:/Users/wepie/dcc-mcp/.../server_with_gui.py`

不要再写旧路径 `C:/Users/wepie/houdini-mcp/...`。

## 4. 健康检查

可执行：

```bash
python verify_codex_setup.py
```

用于检查配置、状态文件和基础进程状态。

## 5. 日志与状态文件

日志目录（示例）：
- `C:/Users/wepie/.mcp_logs/houdini-mcp-daemon/`
- `C:/Users/wepie/.mcp_logs/houdini-mcp-bridge/`

状态目录（示例）：
- `C:/Users/wepie/.codex/mcp/houdini-mcp/`
- `C:/Users/wepie/.codex/mcp/maya-mcp/`
- `C:/Users/wepie/.codex/mcp/blender-mcp/`
- `C:/Users/wepie/.codex/mcp/substance-designer-mcp/`

每个状态目录包含：
- `.running.lock`
- `ws_port.json`
- `ws_port_<pid>.json`

## 6. 常见问题

1. 客户端连不上：
- 先用统一面板测试连接。
- 检查客户端配置是否指向 `dcc-mcp` 路径。

2. 显示在线但调用失败：
- 先“清理旧进程”，再“重启 MCP 服务器”。

3. 面板关闭后服务中断：
- 属于异常情况，先看对应 daemon 日志，再重启该 DCC 的 MCP 服务。

## 7. GUI 自动弹出控制

需要自动弹出时设置环境变量：

```
DCC_MCP_AUTO_OPEN_GUI=1
```

默认不自动弹出。
